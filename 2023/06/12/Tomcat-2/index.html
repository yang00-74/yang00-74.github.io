Tomcat 9.0 源码分析(2)-连接器 Connector 的启动

Connector 连接器启动流程

在 Tomcat源码分析(1)-结构组成与核心组件 中笔者大致分析了 Tomcat 的组成，Connector 连接器是直接和底层 socket 交互的重要组件。 在 Tomcat 的启动过程中，Connector 连接器的初始化及启动流程如下图所示，可以看到主要流程分为 3 个步骤：

  1. 解析配置文件，创建 Connector 连接器对象
  2. 初始化 Connector 连接器各个构件，完成端口绑定监听
  3. 启动 Connector 连接器，开始监听 socket 连接

源码分析

Connector 连接器对象的创建

  1. Tomcat 服务器的启动一般是通过 startup.sh 脚本，另外也可以直接以 jar 包形式启动，二者实际流程差异不大。本文为方便分析，以 Tomcat#main() 方法开始，其实现如下所示，可以看到主要流程如下：

    1. 首先新建抽象整个服务器的 Tomcat 对象
    2. 调用 Tomcat#init() 方法完成服务器初始化
    3. 调用 Tomcat#start() 方法启动服务器各个组件

  2. Tomcat 对象的创建很简单，不需要关注，我们直接从 Tomcat#init() 方法开始分析。这个方法的主要逻辑是新建 Catalina 对象，并通过该对象的 Catalina#load() 方法去加载 xml 配置文件，完成关键组件对象的创建

  3. Catalina#load() 方法源码如下，最关键的操作可以分为两步：

    1. 调用 Catalina#createStartDigester() 方法创建 Digester 对象，该对象中会完成关键组件规则定义，比如指定 Server 的实现类为 StandardServer，并指定其属性注入的反射方法
    2. 调用 Digester#parse() 方法解析 server.xml 文件，完成核心类对象的创建
    3. getServer().init() 调用 StandardServer#init() 开始服务器初始化

  4. Catalina#createStartDigester() 方法的实现如下，可以看到核心就是创建了 Digester，并为其添加了解析 server.xml 文件的规则引擎，这些规则将在解析动作触发时完成关键对象的创建

  5. 一个典型的 server.xml 文件结构如下，在解析 xml 文件过程中会依次创建 StandardServer、StandardService、Connector 对象，本文关注的是连接器的 Connector，故以连接器对象为重点分析

  6. Connector 的构造方法如下，该部分在Tomcat源码分析(1)-Connector 连接器 中也有提及，该部分涉及的其他关键对象的创建参考链接即可，此处不再赘述，至此 Connector 连接器对象的创建告一段落

Connector 连接器初始化

  1. 在上一节步骤3 Catalina#load() 方法的逻辑中，最后一步是通过代码 getServer().init() 调用 StandardServer#init() 方法开始服务器初始化，本节就从此处开始分析。实际上StandardServer#init() 方法是其父类 LifecycleBase 生命周期基类的实现，最终调用到的是 StandardServer#initInternal() 方法。这里是服务器开始初始化的入口，结合 Tomcat 的组件结构，不难看出此处的关键其实是方法最后一部分，遍历 Server 中包含的 Service 对象，依次调用其初始化方法

  2. Service 的实现类为 StandardService，与 Server 类似，Service#init() 最终其实会调用到 StandardService#initInternal() 方法，可以看到这里比较关键的以下两个操作：

    1. engine.init() 调用容器的初始化方法进行容器初始化，本文暂不分析这部分
    2. 遍历 Connector 连接器列表，依次初始化各个连接器

  3. Connector#init() 还是老套路，最终调用到 Connector#initInternal() 方法。这里最重要的逻辑有两处：

    1. 创建 CoyoteAdapter 对象作为 Adapter 适配器，并通过 protocolHandler.setAdapter(adapter) 使得协议处理器能够持有适配器对象
    2. 通过 protocolHandler.init() 调用到协议处理器的初始化方法，最终将调用到 Http11NioProtocol#init() 方法

  4. Http11NioProtocol#init() 方法的重要实现其实是其父类 AbstractProtocol#init() 方法，可以看到这里会通过 endpoint.init() 调用到端点 NioEndpoint#init() 方法

  5. NioEndpoint#init() 方法其实也是父类 AbstractEndpoint#init() 实现，此处的关键如下：

    首先调用 AbstractEndpoint#bindWithCleanup()进入绑定监听端口的流程，在这个方法里再调用 AbstractEndpoint#bind() 进入子类实现的实质性绑定处理，这里其实就是调用 NioEndpoint#bind() 方法

  6. NioEndpoint#bind() 方法如下，可知关键逻辑是调用 NioEndpoint#initServerSocket() 方法完成服务端的端口监听，至此 Connector 连接器的初始化完毕

Connector 连接器启动

  1. 此时回到2.1 节步骤1，可知道服务器的启动其实是由 Tomcat#start() 方法调用触发的，此处最主要的逻辑是调用 StandardServer#start() 方法启动服务器

  2. StandardServer#start() 方法其实是基类方法，最终的逻辑在 StandardServer#startInternal() 方法中。这里与服务器初始化的流程类似，同样是组件一级一级往下启动，最终将调用到 Connector#startInternal() 方法

  3. Connector#startInternal() 方法的重要逻辑是调用 Http11NioProtocol#start() 启动协议处理器

  4. Http11NioProtocol#start() 方法其实是父类实现 AbstractProtocol#start()，这里需要关注的有两个点：

    1. 调用 NioEndpoint#start() 方法使端点开始接收连接并处理 socket 数据，这个方法其实继承自父类 AbstractEndpoint#start() 方法
    2. 启动周期线程池，把 Http11NioProtocol#startAsyncTimeout() 方法包装成任务扔进池子中。需注意，Http11NioProtocol#startAsyncTimeout() 方法其实也会开启每秒 1 次的周期任务，主要用于检查等待处理的 Processor 是否超时，此处是 DeferredResult 等异步请求超时处理的关键

  5. AbstractEndpoint#start() 方法的核心是调用子类的实现，在本文中也就是NioEndPoint#startInternal() 方法

  6. NioEndPoint#startInternal() 方法的关键操作是两步：

    1. 创建 Poller 轮询器，并将其放在另一个线程中启动
    2. 调用父类 AbstractEndpoint#startAcceptorThread() 方法启动连接接收器，开始接受外部连接

  7. 首先来看下连接接收器 Acceptor#run()的处理流程，可知其关键步骤如下：

    1. 如果端点 EndPoint 处于运行状态，则一直 while 循环不断接收新连接
    2. 实际接收连接的操作由 endpoint.serverSocketAccept() 调用 NioEndpoint#serverSocketAccept() 方法完成
    3. 通过 endpoint.setSocketOptions(socket) 设置监听新建的 socket 上的读写事件，此处实际调用 NioEndpoint#setSocketOptions() 方法

  8. NioEndpoint#setSocketOptions() 方法非常关键，核心步骤如下：

    1. 使用新建立的 socket 连接创建 NioChannel 对象，并将其包装到 NioSocketWrapper 对象中
    2. 通过 poller.register(channel, socketWrapper) 调用轮询器 Poller#register() 方法将 Channel 注册到 Selector 上，这样就能通过 Selector 实现监听连接上的读写事件

  9. Poller#register() 的处理如下，可以看到注册并不是直接完成的，而是创建了一个 PollerEvent 异步任务，并将其添加到任务队列中

  10. 任务队列中任务的处理其实在 Poller#run 方法中，可以看到轮询器的关键操作如下：

    1. 每个 while 循环周期首先调用 Poller#events() 方法优先处理队列中的异步任务
    2. 调用 Selector#selectedKeys() 获取有读写事件的连接，调用 Poller#processKey() 方法进行分发处理，这部分本文暂不深入分析

  11. Poller#events() 方法会遍历任务队列，取出其中的任务依次执行，也就是会调用 PollerEvent#run() 方法

  12. PollerEvent#run() 方法如下，其处理逻辑可以分为两部分，至此 Connector 连接器的启动全部完成，可以接收网络连接进行 socket 数据读取了

    1. 对于 OP_REGISTER 事件类型，主要处理为将 Channel 注册到指定的 Selector 上，并为其指定事件监听位为 SelectionKey.OP_READ
    2. 对于其他事件类型，则主要进行关闭 socket、修改 Channel 的事件监听位等处理
